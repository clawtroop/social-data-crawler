## 各平台判定依据

### 1. Wikipedia

- 输出文件：`output/api_completeness_retest_20260329/crawl_out/records.jsonl`
- 关键字段正常：
  - `title = Large language model`
  - `summary` 非空
  - `metadata.pageprops.wikibase-shortdesc = Type of machine learning model`
  - `plain_text` 为完整正文
  - `chunks.total_chunks > 0`

判定：当前 `wikipedia/article` 采集完整。

### 2. arXiv

- 主执行失败信息：
  - `All backends exhausted for https://arxiv.org/abs/1706.03762 (tried ['api'])`
- 直接请求 `http://export.arxiv.org/api/query?id_list=1706.03762` 时出现 `429`
- HTTP 补测成功，但结果退化明显：
  - `metadata.title = open search`
  - `plain_text = ""`
  - `abstract = ""`
  - `chunks = []`

判定：当前 `arxiv/paper` 无法稳定拿到完整论文摘要数据，API 路径受限，HTTP 回退也不足以补齐核心字段。

### 3. Amazon

- 输出标题：`Amazon.com : mechanical keyboard`
- `plain_text` 中可见多条搜索结果信息：
  - 商品标题
  - 评分
  - 购买量
  - 价格
  - 配送信息
- `chunks.total_chunks = 4`

判定：当前 `amazon/search` 可采到较完整搜索结果正文，达到本轮“搜索页数据完整性”要求。

备注：归一化字段仍偏少，主要价值体现在 `plain_text/markdown/chunks`，不是结构化商品字段。

### 4. Base

- 输出虽为 `success`，但核心返回内容为：

```text
You are using a deprecated V1 endpoint, switch to Etherscan API V2
```

- `structured.rpc_result` 也是上述提示，而非合约源码或合约元数据

判定：当前 `base/contract` 接口链路已失效，数据不完整。

### 5. LinkedIn

- `company/microsoft` 主执行失败：
  - `403 Forbidden`
  - `All backends exhausted for https://www.linkedin.com/company/microsoft/ (tried ['api'])`
- `search/company` 补测虽然返回 `success`，但实际命中登录页：
  - `metadata.description = Login to LinkedIn...`
  - `plain_text` 为登录表单文本
  - `result_count = 0`

判定：当前 LinkedIn 会话不可用于有效采集，平台数据不完整。

## 总结

本次 5 平台复测结论如下：

- 完整：`Wikipedia`
- 基本完整：`Amazon(search)`
- 不完整：`arXiv`、`Base(contract)`、`LinkedIn`

按“5 个平台 API 数据是否采集完整”的口径，当前实现不能判定为整体完整，主要阻塞点是：

1. `arXiv` API 访问受限，且新管线未从 `api` 自动切到 fallback
2. `Base` 合约数据仍调用已废弃的 Basescan V1 接口
3. `LinkedIn` 当前会话已不能支撑有效数据抓取，company API 403，search 退化到登录页
