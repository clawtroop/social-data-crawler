# social-data-crawler Skill 使用体验优化方案

日期：2026-03-29

## 背景

当前 `social-data-crawler` 在部分任务上虽然已经具备多后端抓取能力，但真实使用体验仍然不够“代理优先”：

- agent 会把“是否切换到 http / 浏览器态”当成后续建议抛给用户
- 新管线对部分 `api` 平台存在固定后端调用，导致自动降级链没有完整生效
- `backend_routing.json` 与各平台 adapter 的默认后端定义存在不一致
- 用户真正想要的是“给任务目标，工具自动选最稳的抓法并尽量拿全”，而不是在采集中途反复确认技术路径

## 本次优化目标

把 skill 的默认行为改成：

1. agent 自动主导采集链路，不把后端切换决策抛给用户
2. 默认按能力链自动降级：
   - 能用 `api` 就先 `api`
   - `api` 失败就自动转 `http`
   - `http` 再失败就自动转浏览器态
   - 浏览器态优先 `playwright`，必要时再 `camoufox`
3. 只有在以下场景才打断用户：
   - 需要账号登录、验证码、人机验证
   - 页面需要用户授权或人工确认
   - 已经穷尽抓取链路，仍然拿不到可用内容
4. agent 对外汇报应以“已自动尝试了什么、当前结果如何、还缺什么”表述，而不是“你要不要我继续试”

## 目标执行策略

### 1. 技术层

统一自动路由策略：

- `wikipedia` / `arxiv` / `base`：`api -> http -> playwright`
- `amazon` / `generic`：`http -> playwright -> camoufox`
- `linkedin(profile/company/search/job)`：`api -> playwright -> camoufox`
- `linkedin(post)`：`playwright -> camoufox`

规则：

- 未显式传 `--backend` 时，必须允许自动后端升级
- 仅当用户显式传 `--backend` 时，才固定单一后端
- `api` 平台在新管线中也必须保留自动 fallback

### 2. Skill 文档层

Skill 文档需要明确写死以下行为准则：

- “agent 内部自动决策抓取后端”
- “不要把 API/HTTP/浏览器切换作为待用户确认项”
- “抓取不完整时，优先自动升级后端补抓，再对用户汇报结果”
- “用户只负责必要的人机交互，不负责技术路线选择”

### 3. 对用户的话术层

默认输出应该改成这种风格：

> 已写入目标目录。当前内容来自首轮抓取；检测到正文可能未完整展开，我会继续按 `api -> http -> 浏览器态` 自动补抓，并在同一路径覆盖更新结果。只有遇到登录、验证码或权限阻断时才会再提示你介入。

不应该再输出这种风格：

> 如果你要，我下一步可以继续深挖完整页面……
> 你一句话，我继续。

## 验收标准

满足以下条件才算完成：

1. 新管线下，默认运行可自动走完整 fallback 链
2. 显式 `--backend` 时仍保持强制指定行为
3. `backend_routing.json` 与各平台 adapter 默认后端一致
4. Skill 文档中明确禁止把后端切换决策交给用户
5. 测试覆盖：
   - 路由配置
   - 新管线默认不锁死 `api`
   - `api` 失败后可继续 `http / browser`

## 预期收益

- 用户不再承担抓取技术决策
- 公网文档、公开页面、半动态页面的成功率更稳定
- skill 行为更像“任务执行器”而不是“建议生成器”
- 后续把 Notion、帮助中心、文档站等页面抓成完整 Markdown 时，整体体验会更顺
