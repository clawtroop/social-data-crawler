# 联调测试指导清单

本文档用于今天的联调执行，不区分“单测 crawler”或“单测平台”，直接围绕完整链路：

`本地采集 -> 结构化 -> 提交 -> 复爬/评估 -> 结算`

目标是同时验证三件事：

- `social-data-crawler` 能否稳定产出可提交的数据
- Platform Service API 能否正确接收、分发、评估、结算
- 实际实现是否符合协议文档预期

## 1. 测试对象

本次联调涉及三部分：

- 本地采集工具：`D:\kaifa\clawtroop\social-data-crawler`
- 平台接口文档：`http://101.47.73.95/swagger/index.html`

## 2. 鉴权与基础约定

### 2.1 鉴权方式

受保护接口统一使用 Bearer Token：

```http
Authorization: Bearer PLATFORM_SERVICE_JWT_SECRET_TEST
```

注意：

- 不是传裸 token
- Swagger `Authorize` 中也按 `Bearer PLATFORM_SERVICE_JWT_SECRET_TEST` 填

### 2.2 统一记录要求

每个问题都要保留以下信息：

- 接口名
- 请求时间
- 请求参数
- 响应状态码
- 响应体
- `request_id`
- 关联资源 ID，例如 `dataset_id`、`submission_id`、`task_id`、`epoch_id`

## 3. 联调总链路

今天的测试按这条业务链路理解即可：

1. 创建或选择一个 `DataSet`
2. 用 `social-data-crawler` 产出 `cleaned_data` 和 `structured_data`
3. 调用 Core 接口提交 `submission`
4. 平台进行占位、去重和状态管理
5. 平台创建或分发 `repeat-crawl task`
6. 平台创建或分发 `evaluation task`
7. Miner / Validator 完成任务并上报结果
8. Epoch 结算，得到最终 `confirmed / rejected` 结果

## 4. 测试准备

开测前先统一准备：

- 可访问 Swagger UI
- 协议文档可查看
- 本地 `social-data-crawler` 可运行
- 本地准备至少 4 类样本数据

建议样本：

- 正常样本：字段完整，页面可抓，结构化合理
- 重复样本：相同内容重复提交，或相同内容不同 URL
- 脏样本：缺字段、错字段类型、空内容
- 边界样本：404 页面、登录页、动态页、变化页

## 5. 完整测试清单

## 5.1 基础健康检查

目标：确认服务活着、依赖就绪、鉴权边界正常。

需要测试：

- `GET /healthz`
- `GET /readyz`
- 不带 token 调公开接口
- 不带 token 调受保护接口
- 带 token 调受保护接口
- 带错误 token 调受保护接口

通过标准：

- 健康接口返回正常
- 公开接口可匿名访问
- 受保护接口未授权时返回 `401` 或 `403`
- 带正确 Bearer Token 后可正常访问

## 5.2 DataSet 管理链路

目标：确认测试用数据集能创建、能激活、能作为后续 submission 的接收目标。

需要测试：

- `GET /api/core/v1/datasets`
- `POST /api/core/v1/datasets`
- `GET /api/core/v1/datasets/{id}`
- `POST /api/core/v1/datasets/{id}/activate`

重点检查：

- 正常创建
- 必填字段缺失是否拒绝
- 字段类型错误是否拒绝
- 重复 `id` 是否冲突
- 激活前后 `status` 是否正确变化
- 未激活 DataSet 是否禁止提交

协议一致性重点：

- 协议强调 `dedup_fields` 和 `url_patterns`
- 如果 Swagger 请求模型里没有这些字段，或创建后行为不体现这些规则，要单独记录

## 5.3 本地 crawler 数据产出链路

目标：确认本地工具可以生成可用于平台联调的数据。

需要验证：

- 是否能抓到目标网页
- 是否能产出 `cleaned_data`
- 是否能产出 `structured_data`
- 是否符合目标 DataSet 的 schema
- 遇到登录页、动态页、404、异常页时表现如何

重点产出：

- 至少一份正常样本
- 至少一份重复内容样本
- 至少一份缺字段样本
- 至少一份边界或异常样本

### 5.3.1 当前 crawler 到 submission 的实际映射约定

当前 `social-data-crawler` 的标准输出不是平台 `submission` 请求体，而是 agent 友好的标准 record。

联调时统一按下面规则投递到 `POST /api/core/v1/submissions`：

| crawler record 字段 | submission entry 字段 | 当前约定 |
| --- | --- | --- |
| `canonical_url` | `url` | 直接映射 |
| `plain_text` | `cleaned_data` | 当前默认使用 `plain_text`，不使用 `markdown` |
| `structured` | `structured_data` | 直接映射，不自动合并顶层兼容字段 |
| `crawl_timestamp` | `crawl_timestamp` | 优先使用 record 自带字段；若缺失，则回退到同目录 `run_manifest.json.generated_at` |

说明：

- 当前 crawler 输出里常见字段有 `plain_text`、`markdown`、`structured`、`chunks`、`artifacts`
- 平台 submission 只接收 `url`、`cleaned_data`、`structured_data`、`crawl_timestamp`
- 因此 `chunks`、`artifacts`、`metadata` 等字段本次不直接提交给平台

可直接使用新增命令导出平台请求体：

```bash
python -m crawler export-submissions \
  --input ./out/records.jsonl \
  --output ./out/submissions.json \
  --dataset-id ds_posts
```

如果当前 `records.jsonl` 所在目录没有 `run_manifest.json`，或者你想显式指定统一时间戳，则补：

```bash
python -m crawler export-submissions \
  --input ./out/records.jsonl \
  --output ./out/submissions.json \
  --dataset-id ds_posts \
  --generated-at 2026-03-29T10:00:00Z
```

导出的 `submissions.json` 可直接作为 Swagger 中 `CreateSubmissionsRequest` 的 body。

## 5.4 提交与去重链路

目标：确认采集结果进入平台后，能形成 `pending` 记录，并正确处理占位与去重。

需要测试：

- `GET /api/core/v1/url-occupancies/check`
- `GET /api/core/v1/url-occupancies`
- `GET /api/core/v1/url-occupancies/{datasetId}/{urlHash}`
- `POST /api/core/v1/submissions`
- `GET /api/core/v1/submissions`
- `GET /api/core/v1/submissions/{id}`

重点检查：

- 正常 submission 创建成功
- 创建后状态是否为 `pending`
- URL 是否形成占位
- 同一 URL 重复提交是否被拦截
- 相同内容不同 URL 是否被拦截
- `cleaned_data`、`structured_data`、`crawl_timestamp` 是否正确保存

协议一致性重点：

- 协议要求核心去重基于 `dedup_fields`
- 如果当前实现只按 URL 去重，要记录为“实现与协议不一致”

## 5.5 Miner 链路

目标：确认 Miner 在平台中的在线状态、准入与提交前检查流程可用。

需要测试：

- `POST /api/mining/v1/miners/heartbeat`
- `GET /api/mining/v1/miners/online`
- `POST /api/mining/v1/miners/preflight`

重点检查：

- heartbeat 后是否进入在线列表
- 是否返回 credit、在线状态等信息
- preflight 是否返回 `allowed / remaining / challenge`
- 达到限制后是否明确拒绝

协议一致性重点：

- 协议里有信用分、提交上限、PoW 概率、IP 衰减
- 如果接口上只能看到简单 allow/deny，没有对应行为，要记录“协议能力未完全体现”

## 5.6 PoW 链路

目标：确认提交前挑战流程可走通。

需要测试：

- `POST /api/mining/v1/pow-challenges/{id}/answer`

重点检查：

- preflight 触发 challenge 时能否返回 challenge
- 回答正确是否放行
- 回答错误是否拒绝
- challenge 过期后是否不可用
- challenge 是否只能消费一次

## 5.7 Repeat Crawl 任务链路

目标：确认平台可以为 submission 创建重复抓取任务，并让 Miner 正常 claim / report。

需要测试：

- `POST /api/mining/v1/core-submissions/{id}/repeat-crawl-tasks`
- `POST /api/mining/v1/repeat-crawl-tasks`
- `POST /api/mining/v1/repeat-crawl-tasks/claim`
- `POST /api/mining/v1/repeat-crawl-tasks/{id}/report`

重点检查：

- 能否从 submission 创建 repeat crawl task
- claim 后任务是否绑定到指定 miner
- report 后状态是否正确变化
- 非指定 miner 是否不能 report
- 同一任务是否会被重复 claim

协议一致性重点：

- 协议强调 Step 1 / Step 2 两阶段
- 协议强调排除原始提交者及同 IP
- 如果接口模型有字段但运行不体现，要重点记录

## 5.8 Validator 链路

目标：确认 Validator 可以上线、入 ready pool、领取评估任务。

需要测试：

- `POST /api/mining/v1/validators/heartbeat`

重点检查：

- heartbeat 是否成功
- ready 后是否进入 ready pool
- 未 ready 是否不能领取任务
- 重复 ready 是否幂等

## 5.9 Evaluation 任务链路

目标：确认 Validator 评估任务的创建、领取和回报可走通。

需要测试：

- `POST /api/mining/v1/core-submissions/{id}/evaluation-tasks`
- `POST /api/mining/v1/evaluation-tasks`
- `POST /api/mining/v1/evaluation-tasks/claim`
- `POST /api/mining/v1/evaluation-tasks/{id}/report`

重点检查：

- 能否从 core submission 创建 evaluation task
- validator 是否能正确 claim assignment
- report score 后任务是否完成
- 单 validator 模式是否正常
- peer review 是否可观察
- golden task 是否可观察

协议一致性重点：

- 协议里 Phase B、Peer Review、Golden Task 很关键
- 如果只有基础 create/claim/report 壳子，但没有这些策略行为，也要记录

## 5.10 Validation Result 链路

目标：确认 Core 侧验证结果写入能力正常，并能作为结算输入。

需要测试：

- `POST /api/core/v1/validation-results`
- `GET /api/core/v1/validation-results`
- `GET /api/core/v1/validation-results/{id}`

重点检查：

- 能否正确写入 `score / verdict / comment`
- `idempotency_key` 是否幂等
- 非法 `score` 或 `verdict` 是否被拒绝

## 5.11 Epoch 与结算链路

目标：确认数据能从 `pending` 真正进入最终结算状态。

需要测试：

- `GET /api/core/v1/epochs`
- `GET /api/core/v1/epochs/{epochID}`
- `POST /api/core/v1/epochs/{epochID}/settle`
- `GET /api/mining/v1/epochs/{id}/snapshot`
- `GET /api/mining/v1/epochs/{id}/settlement-results`

重点检查：

- epoch 是否可查询
- settle 前后状态是否正确变化
- submission 是否从 `pending` 变成 `confirmed` 或 `rejected`
- snapshot 和 settlement results 是否返回 miner / validator 聚合结果
- 已结算 epoch 是否禁止重复结算

## 7. 协议一致性专项

这是今天最重要的检查项，必须单独整理。

### 7.1 dedup_fields

协议要求按内容字段去重，不是只按 URL 去重。

必须测试：

- 同 URL 重复提交
- 相同内容不同 URL 提交
- 不同内容相同域名提交

### 7.2 url_patterns

协议要求 URL 不匹配规则时要拒绝。

必须测试：

- 合法 URL
- 域名正确但路径不合法 URL
- 完全不相关 URL

### 7.3 状态流转

协议要求核心状态为：

- `pending`
- `confirmed`
- `rejected`

必须测试：

- 提交后是否先进入 `pending`
- 结算后是否进入最终状态
- 最终状态是否和评估结果一致

### 7.4 refresh_interval

协议里明确提到刷新机制和历史版本。

必须检查：

- Swagger 和实际接口是否已有 refresh 相关行为
- 过期数据是否会重新开放提交
- 是否存在 refresh task 相关链路

### 7.5 PoW / 信用分 / IP 衰减

协议写得很细，联调时必须确认：

- preflight 是否体现 PoW
- credit 是否真的参与限制
- 同 IP 多 miner 是否有衰减迹象

### 7.6 Repeat Crawl 两阶段

必须确认：

- 是否存在 Step 1 / Step 2
- Step 1 不一致时是否真的生成 Step 2
- 是否有 phase A result

### 7.7 Golden Task / Peer Review

必须确认：

- 是否可以被实际观察到
- 是否只是文档有、接口无
- 是否接口有、逻辑未生效

## 8. 今日推荐执行顺序

建议按下面顺序跑，避免先做复杂链路再发现基础没通：

1. 健康检查与鉴权
2. 创建并激活一个测试 DataSet
3. 用 crawler 生成一份正常样本
4. 创建 submission 并查询状态
5. 测重复提交和非法提交
6. 跑 miner heartbeat / preflight / pow
7. 跑 repeat-crawl create / claim / report
8. 跑 validator heartbeat / ready / evaluation claim / report
9. 跑 epoch settle
10. 汇总协议不一致点

## 9. Bug 记录模板

每个问题统一按这个格式记录：

- 接口：`POST /api/core/v1/submissions`
- 场景：相同内容不同 URL 重复提交
- 前置条件：dataset 已激活
- 请求：关键参数摘要
- 协议预期：应按 `dedup_fields` 拒绝
- 实际结果：返回 `201`，生成第二条 `pending`
- 结论：与协议不一致
- 证据：响应体、`request_id`、`submission_id`、时间
