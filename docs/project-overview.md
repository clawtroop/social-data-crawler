# Social Data Crawler 项目概览

## 1. 项目定位

Social Data Crawler 是一个面向 Agent 的本地化结构化爬虫项目，目标不是保存原始页面，而是稳定地产出可复用的标准记录。项目支持 Wikipedia、arXiv、Amazon、Base、LinkedIn 五类数据源，并将抓取与增强拆成清晰阶段，便于重跑、恢复和二次处理。

核心处理链路：

发现 → 抓取 → 提取 → 规范化 → 增强 → 写出

## 2. 核心能力

| 能力 | 说明 |
| --- | --- |
| 多平台采集 | 统一处理百科、论文、电商、链上、职场社交数据 |
| 多后端抓取 | 支持 API、HTTP、Playwright、Camoufox 四类抓取后端 |
| 结构化输出 | 输出标准记录、错误记录、摘要、运行清单、制品目录 |
| 会话复用 | 支持导入 Cookies 或浏览器状态，并持久化平台会话 |
| 分阶段执行 | 支持仅抓取、仅增强、抓取加增强三种执行模式 |
| 故障恢复 | 通过状态字段、错误码、下一步动作字段指导重试 |

### 2.1 参考项目落地情况

项目中确实已经落地了一部分参考能力，但不是所有被提到的项目都以“直接接入依赖或完整复刻框架”的方式实现。更准确的情况如下：

| 参考项目 | 当前状态 | 已落地内容 |
| --- | --- | --- |
| Playwright | 已实现 | 已实现浏览器上下文管理、登录态持久化、页面等待、截图、HTML 抓取与浏览器抓取后端 |
| Camoufox | 已实现 | 已作为高风险页面的浏览器后端接入，并纳入 LinkedIn、Amazon 等场景的回退链路 |
| Trafilatura | 已实现 | 已用于网页正文抽取，优先服务文章型 HTML 内容的 Markdown 提取 |
| Unstructured | 已实现 | 已用于 PDF 与富文档分块，输出 `document_blocks` 等文档结构结果 |
| Firecrawl | 部分借鉴 | 已借鉴 URL 到干净内容的抽取思路、失败回退思路与清洁输出目标，但未直接接入 Firecrawl 服务或 SDK |
| Crawl4AI | 部分借鉴 | 已借鉴面向 AI 消费的 Markdown / JSON 风格输出与统一结果契约，但未直接接入 Crawl4AI 框架 |
| Scrapy | 部分借鉴 | 已借鉴分层流水线、重试回退、输出规范与批处理执行思路，但未完整实现 Scrapy 式调度、去重、限流体系 |

如果只看“代码中有明确实现并能直接定位到模块”的参考项，优先级最高的是 `Playwright`、`Camoufox`、`Trafilatura`、`Unstructured`。

## 4. 数据流总览

文件树下面如果只保留一条主线，这个项目的真实工作方式可以概括为：

输入记录进入 → 解析执行模式 → 生成标准 URL → 准备会话与抓取后端 → 抓取原始内容 → 提取正文与结构 → 规范化为统一记录 → 按字段组增强 → 写出结果与恢复信息

三种执行模式分别对应这条主线的不同截面：

| 模式 | 实际经过的流程 |
| --- | --- |
| crawl | 输入记录 → URL 发现 → 抓取 → 提取 → 规范化 → 写出 |
| enrich | 已有记录 → 字段组选路 → 增强 → 写出 |
| run | 输入记录 → URL 发现 → 抓取 → 提取 → 规范化 → 增强 → 写出 |

## 5. 阶段一：输入进入系统

系统从 JSONL 输入开始，每一行代表一条待处理记录。记录至少要提供平台、资源类型，以及能够唯一定位目标资源的关键标识。命令层在这一阶段同时确定输出目录、是否复用历史结果、是否严格失败退出、是否强制指定后端、是否加载模型配置。

| 必备信息 | 作用 |
| --- | --- |
| platform | 决定走哪个平台规则 |
| resource_type | 决定同一平台下的资源分支 |
| 资源标识字段 | 用于后续拼装标准 URL，例如标题、论文编号、商品编号、地址、公开标识等 |

这一阶段的目标不是抓数据，而是把“原始输入”整理成“可调度任务”。

## 6. 阶段二：发现与标准化定位

进入正式抓取前，系统会先做 URL 发现。它会根据平台和资源类型，从模板配置中找出标准主地址，并把输入中的字段别名映射为统一字段，再拼出可重放的 `canonical_url`。

这一阶段还会顺手生成一些可补抓的附加资源地址，例如 arXiv 论文的 PDF 地址。也就是说，系统并不是只知道“主页面在哪”，而是会提前知道“哪些补充材料也值得抓”。

| 发现阶段产物 | 含义 |
| --- | --- |
| canonical_url | 当前记录的标准主地址 |
| fields | 参与 URL 拼装后的规范字段 |
| artifacts | 可选的次级资源地址，例如 PDF |

## 7. 阶段三：会话准备与抓取策略决策

拿到标准地址后，系统不会立刻请求页面，而是先决定“用什么方式抓”。这一层决策主要由平台规则、资源类型、是否需要登录态、是否显式指定后端共同决定。

| 抓取后端 | 适用场景 |
| --- | --- |
| API | 平台存在稳定接口时优先使用，结构化程度最高 |
| HTTP | 适合公开静态页面，成本最低 |
| Playwright | 适合动态页面、需要渲染或需要登录态的页面 |
| Camoufox | 作为更高风控场景下的浏览器补位方案 |

不同平台的起始策略并不相同：

| 平台 | 常见起点 |
| --- | --- |
| Wikipedia | 优先 API |
| arXiv | 优先 API |
| Amazon | 优先 HTTP 或浏览器页面抓取 |
| Base | 优先链上 API 或 RPC |
| LinkedIn | 登录态存在时优先 API，部分资源直接走浏览器 |

对于 `LinkedIn` 这类依赖登录态的平台，系统会在这一阶段检查会话是否存在、是否有效。会话文件会被整理为统一的浏览器 `storage_state`，保存到输出目录下的会话目录中，后续同目录任务可以直接复用。

推荐的人机协同方式是：

auto-browser 获取登录态 → 导出会话文件 → crawler 导入并标准化保存 → 后续抓取自动复用

如果缺少会话，会明确返回 `AUTH_REQUIRED`；如果会话已失效，会返回 `AUTH_EXPIRED`。这意味着登录问题被设计成一种可恢复状态，而不是模糊失败。

## 8. 阶段四：原始抓取与制品沉淀

确定后端后，系统开始抓取原始内容。这个阶段的核心目标有两个：第一，尽可能成功拿到源内容；第二，把原始现场保留下来，供后续排查和复用。

抓取结果可能是以下几种形态：

| 原始结果 | 说明 |
| --- | --- |
| JSON 数据 | 通常来自 API，结构稳定 |
| HTML 页面 | 来自 HTTP 或浏览器抓取 |
| 二进制文档 | 例如 PDF |
| 截图 | 浏览器抓取时保留页面快照 |

一旦抓取成功，系统会优先把原始制品写入 `artifacts` 目录，例如接口响应、页面 HTML、截图、抓取元数据、补充 PDF。这样即使后续提取或增强失败，排查时依然有原始上下文可回看。

浏览器抓取还有一个额外动作：如果页面运行过程中登录态被刷新，系统会把最新会话重新写回本地，从而让下一次抓取直接继承更新后的状态。

## 9. 阶段五：内容提取

抓到原始内容后，系统会按内容类型走不同提取路径，而不是统一粗暴转文本。

| 内容类型 | 提取方式 |
| --- | --- |
| API JSON | 直接抽结构字段，再生成可读文本与 Markdown |
| HTML | 先清噪，再识别主内容区，再切分章节与文本块 |
| PDF/文档 | 走文档分块提取，生成 `document_blocks` |

HTML 路径内部又分几步：

| 子步骤 | 作用 |
| --- | --- |
| 清噪 | 去掉导航、广告、页脚、评论、隐藏元素等干扰信息 |
| 主体识别 | 优先使用平台选择器，其次用语义标签和文本密度推断正文区域 |
| 内容压缩 | 在主体区域内继续压掉明显 boilerplate、重复段落，保留标题和列表等结构 |
| 章节拆分 | 按标题层级生成 section 路径 |
| 分块 | 把长正文切成适合后续模型处理的 chunk |
| 元信息提取 | 从页面 meta 中抽标题、描述、图片、canonical 等结构字段 |

这一阶段的最终目标不是“保留整页”，而是沉淀出后续可直接消费的三类核心内容：

| 结果类型 | 用途 |
| --- | --- |
| cleaned_html | 主体识别与压缩后的 HTML，中间层调试产物 |
| plain_text | 适合检索、摘要、规则匹配 |
| markdown | 适合保留结构化可读性 |
| document_blocks 或 chunks | 适合文档理解与模型切片处理 |

这里需要明确三层边界：

- `cleaned_html` 是抽取阶段内部的中间 HTML，不是对外主消费字段
- `plain_text` 和 `markdown` 已经是压缩后的 LLM-ready 内容，不再等同于“原页面全文转文本”
- `chunks` 是基于压缩后的内容继续切片生成，不直接从原始 HTML 切

HTML 路径还支持一个显式启用的 CSS 结构化提取入口：

- 默认情况下，HTML 结构字段仍主要来自页面 meta 和现有正文提取链
- 只有命令行传入 `--css-schema <path>` 时，系统才会按给定 schema 用 CSS selector 抽取额外结构字段
- 这条能力是 opt-in 的增强入口，不改变默认提取行为

同样，项目现在支持两条分离的 LLM schema 路径：

- `--extract-llm-schema <path>`：在提取阶段基于压缩后的页面内容补充 `structured`
- `--enrich-llm-schema <path>`：在增强阶段通过 `llm_schema` 字段组补充 `enrichment`

这两条路径共享底层模型执行器，但不共享业务 schema 文件，也不会互相回写对方的数据区域。

## 10. 阶段六：规范化为统一记录

提取完成后，系统会把不同平台、不同资源、不同抓取方式产生的数据，统一装进同一份标准记录里。这个阶段是整个项目最重要的收口层，因为它把多源异构结果压成了统一契约。

标准记录中最关键的字段包括：

| 字段 | 含义 |
| --- | --- |
| platform | 数据来源平台 |
| entity_type | 规范化后的实体类型 |
| resource_type | 原始资源类型 |
| canonical_url | 标准主地址 |
| status | 当前处理状态 |
| stage | 当前阶段 |
| retryable | 是否可重试 |
| error_code | 失败编码 |
| next_action | 建议下一步动作 |
| artifacts | 关联制品列表 |
| metadata | 提取阶段元信息 |
| plain_text | 主体纯文本 |
| markdown | 主体富文本表示 |
| chunks | 基于压缩后主体内容生成的语义切片 |
| document_blocks | 文档块结构 |
| structured | 平台特定结构化字段 |

在这一层里，平台适配器会把平台私有信息写入 `structured`，例如：

| 平台 | 常见结构化结果 |
| --- | --- |
| Wikipedia | 标题、摘要、分类 |
| arXiv | 论文编号、摘要、作者、PDF 文档块 |
| Amazon | 商品编号、标题等页面提取结果 |
| Base | 地址、交易、合约对应的 RPC 结果 |
| LinkedIn | 个人、公司、职位等 Voyager 结构化信息 |

也正因为统一记录已经足够完整，所以它既可以直接作为最终产物，也可以作为后续 `enrich` 阶段的标准输入。

## 11. 阶段七：字段组增强

如果执行的是 `enrich` 或 `run`，系统会在规范化记录之上继续做字段增强。增强不是对整条记录做一次黑箱总结，而是按字段组分别决策、分别执行。

增强阶段会先判断当前记录应该走哪些字段组，再按字段组选择策略：

| 增强策略 | 说明 |
| --- | --- |
| extractive_only | 只依赖规则、查表、正则等确定性方法 |
| generative_only | 直接走模型生成 |
| extractive_then_generative | 先规则抽取，不够确定时再交给模型补全 |

增强结果会被组织成统一结果包，记录字段值、证据、置信度、来源方式、模型信息等内容。这样做的好处是：增强不是覆盖原始数据，而是在标准记录上叠加一层可追溯的解释信息。

| 增强结果组成 | 说明 |
| --- | --- |
| generated_fields | 本次字段组产出的字段 |
| evidence | 该结果依赖了哪些源字段或证据 |
| confidence | 当前结果的可信度 |
| source_fields | 实际参与路由和生成的输入字段 |
| unsupported_reason | 为什么该字段组当前无法执行 |
| model 或 model_config | 使用了什么模型与配置 |

## 12. 阶段八：写出结果、恢复与复跑

所有阶段完成后，系统会把成功结果、失败信息和运行摘要一起写出，形成一套适合 Agent 后续继续消费的运行目录。

| 输出文件 | 作用 |
| --- | --- |
| records.jsonl | 成功记录主文件，也是后续增强与复跑的事实来源 |
| errors.jsonl | 失败记录、错误码与建议动作 |
| summary.json | 本次任务的整体摘要 |
| run_manifest.json | 本次命令、输入输出路径、后端和运行信息 |
| artifacts 目录 | 保留排查和复用所需的原始制品 |

恢复路径也围绕这几份文件设计：

| 场景 | 推荐动作 |
| --- | --- |
| 抓取部分失败 | 先看 `summary.json`，再看 `errors.jsonl` 定位失败原因 |
| 登录态缺失或过期 | 重新导入或刷新会话，再基于同目录复跑 |
| 抓取成功但增强缺失 | 直接对既有 `records.jsonl` 重新执行增强 |
| 需要持续累加结果 | 使用恢复或续跑模式继续写入同一输出树 |

这也是项目最适合 Agent 工作流的地方：它把一次抓取任务变成了可恢复、可复用、可增量补跑的结构化数据流水线，而不是一次性的脚本执行。

## 13. 一句话总结

这是一个围绕“数据如何流动”设计的 Agent 友好型爬虫系统：前段负责把多源输入稳定变成可抓取目标，中段负责把原始内容沉淀为统一记录，后段负责把记录继续增强并以可恢复的方式写出。
