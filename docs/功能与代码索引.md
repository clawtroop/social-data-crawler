## 七、URL 发现 Discovery

### 7.1 功能说明

- **`discover-crawl`**：当前唯一对外 discovery CLI。它在「发现」语境下**真实拉取页面**（通过 `FetchEngine`），由适配器 `crawl` 决定下一步；支持**多跳扩展**（调度器 + 去重 + `spawned_candidates`）。可与 **BFS**（`bfs_engine.py`）等组合做更广的链式发现。
- **`map()` / `MapResult`**：当前保留为 discovery adapter **内部能力**，用于把种子或 HTML 转成候选 URL；不再作为独立 CLI 入口暴露给用户或 agent。
- **适配器**（`discovery/adapters/`）：与 `platforms/` **两套体系**——发现层只管「从种子推更多 URL」，主流水线管「抓全站并结构化」；平台名字符串应对齐。
- **状态与存储**：调度器、前沿队列、visited、checkpoint、occupancy 等（见 `state/`、`store/`）。

### 7.2 Discover 发现功能详解（通俗完整版）

#### 7.2.1 这套功能是解决什么问题的？

日常「爬一条 URL、抽正文、再 enrich」解决的是：**我已经知道要哪一页**。  
**Discover** 解决的是：**我只有模糊起点，需要系统在站点里帮我把「下一步该抓哪些 URL」列出来**。

可以把它理解成两层能力，但对外只开放后一层：

| 比喻 | 对应能力 |
| --- | --- |
| **内部候选映射** | `map()`：适配器内部把种子 +（可选）HTML 转成候选 URL，不作为 CLI 单独暴露。 |
| **带着爬虫边走边分叉** | `discover-crawl`：用统一抓取引擎**真的去请求页面**，适配器在 `crawl` 里决定**本页抓完后还要不要跟哪些新 URL**，形成**多跳、有深度上限**的发现过程。 |

两者都**不等同于** `run` / `crawl` 主流程：主流程目标是 **canonical 记录 + 正文/chunks + 可选 enrich**；Discover 的目标是 **发现更多 `canonical_url` 与元数据**，输出更轻、更偏「导航/拓线」。

#### 7.2.2 和主流水线 `platforms/` 是什么关系？

- **`crawler/platforms/`**：给 **`crawl` / `run` / `enrich`** 用——声明用什么后端抓、怎么抽正文、怎么归一化、默认 enrich 字段组。
- **`crawler/discovery/adapters/`**：给 **`discover-crawl`** 与内部 discovery 能力用——声明**如何从种子扩展 URL**（`map` / `crawl` / 可选 `normalize_url` + `expand` 供 BFS）。

同一个业务站（例如 `linkedin`）可以两边各有一份适配器：**名字相同，职责不同**，不要混在一个类里。

#### 7.2.3 四个核心「名词」

1. **`DiscoveryRecord`（种子记录）**  
   表示「一次发现的起点」：已有规范 URL，或根据 `references/url_templates.json` + 输入字段**拼出来的规范 URL**（见 `url_builder.build_seed_records`）。  
   - 若输入里已有 `canonical_url`，会带上 `identity` 等上下文。  
   - 若没有，会走 `build_url()`，按平台/资源类型模板生成链接（`TEMPLATE_CONSTRUCTION` 模式）。

2. **`DiscoveryCandidate`（候选）**  
   表示「从种子里长出来的、待继续处理的一个 URL + 平台语义」：包含 `canonical_url`、`resource_type`、`hop_depth`（第几跳）、`discovery_mode`（怎么来的）、`score`（优先级线索）等。  
   `discover-crawl` 的输入行会直接变成 **`hop_depth=0` 的候选**。

3. **`DiscoveryMode`（发现模式）**  
   标记这条候选是**怎么被识别出来的**：例如直接输入、规范模板构造、搜索结果、页面内链接、站点图等（见 `contracts.py` 枚举）。便于下游过滤和调试。

4. **`MapResult`（映射结果）**  
   `map()` 的返回值：`accepted` 里是一批 **`DiscoveryCandidate`**，表示「从当前种子/页面里**认可**的下一步 URL」；还可带 `rejected`、`next_seeds` 等扩展字段。

#### 7.2.4 `discover-crawl` 在代码里怎么走？（多跳 + 真抓取）

CLI：`python -m crawler discover-crawl --input ... --output ...`，常用 **`--max-depth`**、**`--max-pages`** 控制深度与总页数上限。

流程要点（`core/pipeline.py` → `_run_discovery_crawl_pipeline_async`）：

1. 输入 JSONL 每行至少要有 **`url` 或 `canonical_url`**；可选 `platform`、`resource_type`，组装为 **`DiscoveryCandidate`**（`hop_depth=0`，`discovery_mode=DIRECT_INPUT`）。
2. 构造异步 **`fetch_fn`**：内部用 **`FetchEngine.fetch`**，返回与主流程一致的 **legacy dict**（含 `html` 等）。
3. 调用 **`run_discover_crawl(..., adapter_resolver=get_discovery_adapter)`**。  
   当传入 **`adapter_resolver`** 时，走 **`_run_discover_crawl_graph`（图/队列模式）**（见 `discovery/runner.py`）：
   - 用 **`DiscoveryScheduler`** 维护**前沿队列**（下一个该处理谁）。
   - 用 **`InMemoryVisitedStore`** 按 `platform:canonical_url` **去重**，避免死循环。
   - 每一轮从队列取一个候选，调用 **`adapter.crawl(candidate, context)`**，其中 **`context["fetch_fn"]`** 会再去拉当前页。
   - 适配器返回的字典里若有 **`spawned_candidates`**（一批新的 `DiscoveryCandidate`），会 **enqueue**，并受 **`max_depth`**、**`max_pages`** 约束。
4. 输出是多条**带 `fetched` 载荷**的记录（已抓过的页面内容摘要），用于后续分析或再喂给主爬。

**与仅循环种子的区别**：`runner.run_discover_crawl` 在 **未传入 `adapter_resolver`** 时，只会对每个 seed **单次 fetch**（简单模式）；**CLI 路径始终传入解析器**，因此默认是**可分叉的图模式**。

#### 7.2.5 举例：LinkedIn 发现适配器在做什么？

文件：`crawler/discovery/adapters/linkedin.py`（配合 `discovery/normalize/linkedin.py`、`expand/*.py`）。

- **`map`**：  
  - 若 context 里已有 `search_candidates`，可直接升格为实体候选；  
  - 否则解析 HTML 里的 `a[href]` 和正则，抽出 **公司 / 个人 / 职位** 等链接，再交给 `map_search_candidates` 生成 **`DiscoveryCandidate`**。
- **`crawl`**：  
  - 先 **`fetch_fn`** 拉当前页 → 把 `html` 再交给 **`map`**；  
  - 把 `map` 接受的候选 **`hop_depth+1`**，填 **`parent_url`**，作为 **`spawned_candidates`** 返回，供 runner **继续入队**。

这样 **`discover-crawl`** 才能从「一个 URL」扩出「多跳相关 URL」，而不是只停在一页。

#### 7.2.6 BFS `run_bfs_expand` 是什么？和上面什么关系？

文件：`crawler/discovery/bfs_engine.py`。

- 这是**另一条**「广度优先」扩展入口：显式传入 **种子 URL 列表**、**异步 `fetch_fn`**、以及实现了 **`normalize_url` + `expand`** 的适配器。
- **`expand`** 会按实体类型调到 `expand/linkedin_profile.py` 等，从详情页再拉**相关链接**（例如公司页的职位、动态等）。
- 它更适合被**脚本或其它模块调用**，或在后续产品与 `discover-crawl` **组合**；**默认 CLI 的 `discover-crawl` 走的是 runner 的图模式 + `adapter.crawl`**，不是直接等于 `run_bfs_expand`。开发时若要「纯 BFS 拓图」，应阅读 **`bfs_engine.py`** 与对应 **`expand/*.py`**。

#### 7.2.7 相关 CLI 参数（发现专用）

| 参数 | 含义 |
| --- | --- |
| `--max-depth` | 候选的 **最大 hop**（图模式里子候选深度不能超过此值）。 |
| `--max-pages` | **discover-crawl** 里为最多产出/处理多少条记录上限。 |
| `--sitemap-mode` | `include` / `only` / `skip`，影响与站点图相关的策略（若适配器实现中使用）。 |

会话目录：发现流程会在 **`--output` 下创建 `.sessions/`** 供 `FetchEngine` 使用（与主爬一致）。

#### 7.2.8 读完这一节你可以记住的三句话

1. **`discover-crawl`**：当前唯一对外 discovery CLI，负责**真的抓页 + 多跳发现**。  
2. **`map()` / `MapResult`**：仍然存在，但属于**适配器内部候选映射能力**，不再单独对外暴露。  
3. **发现适配器**只管 **URL 拓扑与平台语义**；**平台 `PlatformAdapter`** 管 **全量内容与业务字段**——扩展时两套注册表都要想到。

## 八、数据增强 Enrichment

### 8.1 功能说明

- **`EnrichPipeline`**：按 **字段组**（field group）执行**抽取式**（regex/lookup）与**生成式**（模板 + LLM；可返回 `pending_agent` 供外部填回）。
- **`enrich` 子命令**：跳过 fetch/extract，仅对已有记录做 enrich（见 `input_normalizer`）。

- **Legacy 编排**：`orchestrator.py` 的 `route_enrichment` 供旧 dispatcher 使用；生成式策略与新管线不同。
